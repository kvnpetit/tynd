# API Reference

Every Tynd app has three surfaces:

- **Backend** (`@tynd/core`) ŌĆö imported by `backend/main.ts`, exposes `app.start`, emitters, lifecycle hooks.
- **Frontend RPC** (`@tynd/core/client`) ŌĆö typed proxy to your backend functions via `createBackend<typeof backend>()`.
- **OS APIs** (`@tynd/core/client`) ŌĆö direct bridge from the frontend to the Rust host: dialog, window, clipboard, shell, notification, tray, process, fs, http, websocket, sql, sidecar, terminal, store, os, path, compute, and more.

<Callout type="info">
  **Lite vs full parity.** All OS APIs live in Rust (`packages/host-rs/src/os/`) so both runtimes expose the exact same surface. See [Runtime Modes](/docs/v0.3/runtimes) for JS-runtime differences.
</Callout>

## Backend

- [`@tynd/core`](/docs/v0.3/api/backend) ŌĆö `app.start`, `app.onReady`, `app.onClose`, `createEmitter`, `AppConfig`, `WindowConfig`, menu/tray config.
- [Streaming RPC](/docs/v0.3/api/streaming-rpc) ŌĆö async-generator handlers.

## Web-platform re-exports

`@tynd/core/client` also re-exports the standard Web globals as named exports so `import * as tynd` surfaces them in one namespace alongside the Tynd OS APIs:

```ts
import * as tynd from "@tynd/core/client";

await tynd.fetch(url);
const ws = new tynd.WebSocket(wsUrl);
const hash = await tynd.crypto.subtle.digest("SHA-256", bytes);

await tynd.fs.readText(path);       // Tynd OS API on the same namespace
await tynd.sql.open(dbPath);
```

Available re-exports: `fetch`, `Request`, `Response`, `Headers`, `AbortController`, `AbortSignal`, `ReadableStream`, `WebSocket`, `EventSource`, `crypto`, `URL`, `URLSearchParams`, `TextEncoder`, `TextDecoder`, `atob`, `btoa`, `Blob`, `File`, `FormData`, `structuredClone`, `performance`.

Behavior matches the Web spec in both runtimes. In `lite` they point at Tynd's polyfills; in `full` they point at Bun's native implementations.
